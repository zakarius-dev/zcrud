# Story 4.5 : Sous-listes / relations imbriquées (`ZSubListScreen`, mini-CRUD) & onglets de catégorisation (`ZTabbedList`) — clôture d'E4 (étend FR-6)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

En tant que **développeur intégrant `zcrud` dans une app hôte (DODLP puis lex_douane)**,
je veux **présenter, à l'intérieur d'un écran parent, une SOUS-LISTE d'entités RELIÉES à un parent — un `ZSubListScreen<T>` plein-champ qui EST une `DynamicList` autonome (ses PROPRES recherche / tri / pagination / actions de ligne filtrées `ZAcl` / sélection / corbeille soft-delete-restore), filtrée par la RELATION parent→enfants exprimée dans le contrat NEUTRE (`ZFilter(parentField, eq, parentId)` dans `ZDataRequest`) — ET catégoriser une collection en ONGLETS (`ZTabbedList`) où chaque onglet est une `DynamicList`/`ZListController` avec son propre filtre de catégorie, dont l'ÉTAT (recherche/tri/pagination/sélection/scroll) est PRÉSERVÉ au changement d'onglet**,
afin que **le moteur de liste couvre enfin la capacité « sous-listes/relations, onglets » du §4.2 du PRD (rattachée à FR-6) SANS réécrire la machinerie de liste : `ZSubListScreen` et `ZTabbedList` RÉUTILISENT à 100 % `ZListController` (E4-3) + `DynamicList<T>` (E4-2) + le port `ZListRenderer` injecté (E4-1) + les actions/`ZAcl`/sélection/corbeille (E4-4) — le cœur restant NEUTRE (SM-5 : le chrome des onglets est pur-Flutter Material, aucune `SfDataGrid` dans `zcrud_core`, la seule arête Syncfusion demeure `ZSfDataGridRenderer` dans `zcrud_list`), la relation ne faisant fuiter AUCUN type backend (AD-16 : `ZFilter`/`ZDataRequest` neutres, honorés par l'adaptateur E5)**.

**Contexte produit.** **Cinquième et DERNIÈRE story d'E4** (moteur `DynamicList`, dépend d'E2 ; couvre FR-6..FR-8 · AD-8, AD-11, AD-13, AD-16, AD-2, AD-15). Après cette story, **E4 est complète** → `bmad-retrospective` d'E4. E4-5 assemble les briques posées par E4-1..E4-4 en deux surfaces de composition — la sous-liste reliée et les onglets — **sans nouvelle machinerie de liste** :

- **E4-1 (`done`)** — port **`ZListRenderer`** (`Widget build(BuildContext, ZListRenderRequest, {ZListInteraction? interaction})`) + backend **`ZSfDataGridRenderer`** (**SEULE** arête Syncfusion, `packages/zcrud_list/lib/src/presentation/z_sf_data_grid_renderer.dart`). `ZSubListScreen`/`ZTabbedList` **consomment** le renderer injecté via `ZcrudScope.listRenderer` — ils n'importent JAMAIS `zcrud_list`.
- **E4-2 (`done`)** — **`DynamicList<T extends ZEntity>`** (`StatelessWidget`, dispatch `ZListLayout` = `dataGrid`/`builder`/`custom` + 4 vues d'état) + `ZListRenderRequest.fromSchema`/`deriveColumns`/`ZListColumn`/`ZListRow{id,cells}`. **Réutilisé tel quel** : `ZSubListScreen` rend une `DynamicList<T>` ; chaque onglet de `ZTabbedList` rend une `DynamicList<T>`.
- **E4-3 (`done`)** — **`ZListController<T extends ZEntity>`** (`ChangeNotifier`, `ValueListenable<ZListViewState>`) : détient le `ZDataRequest` courant (filtres/tri/recherche), recherche **sans accents** (`searchable`), moteur in-memory + pagination curseur/repli, `setSearch`/`setFilters`/`setSort`/`refresh`/`loadMore`/`dispose`, `watchMutations` optionnel (`packages/zcrud_core/lib/src/presentation/list/z_list_controller.dart:54`). **Cœur réutilisé** : chaque sous-liste/onglet possède **son** `ZListController`. ⚠️ **`setFilters` REMPLACE** les filtres → la relation parent et le filtre de catégorie doivent être des **`baseFilters` PERSISTANTS** (voir Tâche 1).
- **E4-4 (`done`)** — **`ZRowAction<T>`/`ZResolvedRowAction`** (actions ligne neutres filtrées `ZAcl`, AD-16), **`ZListSelectionController`** (`ChangeNotifier`, `ValueListenable<Set<String>>` keyé par `id` **stable** — bug historique corrigé), **`ZListInteraction`** (pont neutre), corbeille **soft-delete/restore** (`ZRowAction.softDelete`/`restore` + `softDeleteSelected`). **Réutilisé tel quel** dans chaque sous-liste/onglet ; le keying par `id` stable est ce qui rend la sélection survivante au changement d'onglet.

**Ce que E4-5 matérialise :**
1. **`ZSubListScreen<T>`** — un `StatelessWidget`/`StatefulWidget` mince qui, à partir d'un **parent** (`parentField` + `parentId`) et d'un `ZRepository<T>` d'enfants, construit un `ZListController<T>` **seedé** d'un **filtre de relation PERSISTANT** (`ZFilter(parentField, ZFilterOp.eq, parentId)`) et rend une `DynamicList<T>` **complète** (recherche/tri/pagination/actions `ZAcl`/sélection/corbeille). C'est un **mini-CRUD imbriqué** : la sous-liste a **sa propre** vie (recherche/filtres/tri/pagination/actions), **filtrée en permanence** par la relation.
2. **`ZTabbedList`** — un widget d'**onglets de catégorisation** : une liste de `ZListTab` (clé de libellé l10n + filtres de catégorie + fabrique de `ZListController`) rendue en `TabBar`/`TabBarView` **pur-Flutter Material** ; **chaque onglet = une `DynamicList`/`ZListController` indépendante** avec son propre filtre. L'**état de chaque onglet est PRÉSERVÉ** (contrôleurs **non disposés** au switch, `AutomaticKeepAliveClientMixin` sur chaque page) : recherche/tri/pagination/**sélection**/scroll survivent à un aller-retour d'onglet.
3. **Relation parent→enfants NEUTRE** — exprimée **uniquement** via `ZFilter`/`ZDataRequest` (contrat neutre AD-16) ; **aucun** type `cloud_firestore`/`Filter`/`Query`. L'adaptateur backend (E5 `zcrud_firestore`) honore le filtre ; ici, prouvé via un **fake `ZRepository`** in-memory qui applique les filtres.
4. **`baseFilters` PERSISTANTS sur `ZListController`** — extension **additive et rétro-compatible** (défaut `const []`) : les `baseFilters` sont **toujours ANDés** aux filtres utilisateur dans chaque requête, si bien qu'une recherche/un filtre/un tri de la sous-liste (ou de l'onglet) **ne peut jamais** écraser la relation parent ni le filtre de catégorie.
5. **Résolution du report LOW-3 (E4-4)** — sémantique de sélection au filtre/refresh **tranchée** ici (voir *Décisions*).

**Frontière (NE PAS déborder).** E4-5 s'arrête à la **composition de listes** (sous-liste reliée + onglets) **dans `zcrud_core`**, testée avec des fakes. **HORS périmètre** : l'intégration dans l'app réelle (écrans DODLP/lex_douane, navigation) = **E7/E8** ; l'implémentation Firestore de la relation/pagination = **E5** ; le **listing de la corbeille** (voir les supprimés) reste déféré (contrat E2-2 gelé, cf. E4-4) ; l'**export** (E11b/`zcrud_export`). **Ne PAS** dupliquer **E3-3b-2** (le CHAMP d'édition sous-liste inline — voir *Distinction E3-3b-2*).

## Acceptance Criteria

> Tous les ACs sont **testables** : unit `flutter_test` (`baseFilters` composés) et widget `flutter_test` (`ZSubListScreen`/`ZTabbedList` avec **fake `ZRepository`** appliquant les filtres, **fake `ZAcl`**, `FakeEntity implements ZEntity`, layout **`builder`** → **SM-5 exécutable sans `zcrud_list`**). Enjeux headline : **sous-liste filtrée par la relation + mini-CRUD fonctionnel** ; **onglets préservant l'état/sélection au switch** ; **réutilisation d'E4-1..E4-4 sans duplication** ; **SM-5 maintenue** ; **a11y onglets**.

1. **AC1 — `ZSubListScreen<T>` affiche les enfants d'un parent (relation neutre).** À partir de `{repository, parentField, parentId, toRow, schema}`, `ZSubListScreen<T>` construit un `ZListController<T>` seedé de `baseFilters: [ZFilter(parentField, ZFilterOp.eq, parentId)]` et rend une `DynamicList<T>`. Seuls les enfants dont `parentField == parentId` sont affichés. Testable : fake `ZRepository<FakeEntity>` contenant des entités de **deux** parents (`p1`, `p2`) qui **applique** les filtres de `ZDataRequest` ; `ZSubListScreen(parentField:'parentId', parentId:'p1', …)` → seules les lignes de `p1` sont rendues (finder), celles de `p2` absentes.

2. **AC2 — `baseFilters` PERSISTANTS : la relation n'est jamais écrasée (additif rétro-compatible).** `ZListController` accepte `List<ZFilter> baseFilters = const []` ; chaque `ZDataRequest` émis porte `filters: [...baseFilters, ...userFilters]`. Un `setFilters([...])` **utilisateur** compose **en plus** de `baseFilters` (ne le remplace pas) ; `setSearch`/`setSort` conservent `baseFilters`. **Rétro-compatibilité** : `baseFilters` vide ⇒ requêtes **identiques** à E4-3 (tests E4-3 verts, inchangés). Testable (unit) : contrôleur avec `baseFilters:[ZFilter('parentId',eq,'p1')]` + fake repo enregistrant les `ZDataRequest` reçus → `setFilters([ZFilter('status',eq,'open')])` ⇒ requête contient **les deux** filtres ; `setSearch('x')` ⇒ `baseFilters` toujours présent ; contrôleur sans `baseFilters` ⇒ requêtes inchangées.

3. **AC3 — Mini-CRUD complet DANS la sous-liste (réutilise E4-3/E4-4, zéro duplication).** La sous-liste supporte, **en plus** du filtre de relation : (a) **recherche sans accents** (`controller.setSearch` via une barre optionnelle) ; (b) **actions de ligne filtrées `ZAcl`** (E4-4, AD-16) ; (c) **sélection multiple** stable ; (d) **corbeille soft-delete/restore** (`ZRowAction.softDelete`/`restore`). Aucune de ces mécaniques n'est **réimplémentée** — `ZSubListScreen` passe `rowActions`/`selection`/`entityFor`/`actionAclMode` à `DynamicList<T>` et `setSearch` au contrôleur. Testable : (a) taper une recherche masque les enfants non-correspondants **tout en gardant** le filtre parent ; (b) fake `ZAcl` refusant `delete` → action `delete` masquée ; (c) `softDelete` d'un enfant → exclu de la vue (parent-filtrée) après `refresh`, `restore` → réinclus ; (d) sélection de 2 enfants stable au rebuild.

4. **AC4 — `ZTabbedList` : onglets de catégorisation, chaque onglet = une liste indépendante.** `ZTabbedList(tabs: [ZListTab(labelKey, builder), …])` rend un `TabBar` (N onglets) + `TabBarView` où **chaque onglet** construit sa **propre** `DynamicList`/`ZListController` (typiquement avec un **filtre de catégorie** via `baseFilters`). Testable : 3 onglets, chacun catégorisant par un filtre distinct (`status: open|closed|archived`) sur un fake repo → l'onglet actif affiche **son** sous-ensemble ; changer d'onglet affiche le sous-ensemble suivant.

5. **AC5 — État de chaque onglet PRÉSERVÉ au changement d'onglet.** Recherche / tri / pagination (`loadMore`) / **sélection** / position de **scroll** d'un onglet **survivent** à un aller-retour (`onglet A → B → A`) : les `ZListController`/`ZListSelectionController` de chaque onglet ne sont **ni recréés ni disposés** au switch (`AutomaticKeepAliveClientMixin`). Testable (widget) : saisir une recherche dans l'onglet A, sélectionner une ligne, passer à B puis revenir à A → la recherche est **toujours** appliquée, la ligne **toujours** sélectionnée (finders), sans nouvelle requête de chargement (compteur de construction du contrôleur = 1).

6. **AC6 — Le changement d'onglet NE PERD PAS la sélection/scroll (sélections indépendantes par onglet).** Chaque onglet possède **sa propre** `ZListSelectionController` : sélectionner dans A n'affecte pas B ; revenir à A restaure la sélection de A ; la position de scroll de chaque onglet est conservée (KeepAlive). Testable : sélectionner `{a1}` dans A, aller à B (sélection B vide), sélectionner `{b1}` dans B, revenir à A → sélection A = `{a1}` (pas `{b1}`), revenir à B → `{b1}`.

7. **AC7 — Relation parent→enfants NEUTRE (aucune fuite backend, AD-16).** Le filtrage de relation est exprimé **uniquement** via `ZFilter`/`ZDataRequest` ; le nouveau code n'importe **aucun** `cloud_firestore`/`firebase`/`hive` ni `Filter`/`Query`/`Timestamp`. `parentId` est un `Object` opaque (le cœur ne présume pas de sa forme). Testable : garde de pureté (`domain_purity`/`presentation_purity`) verte sur les nouveaux fichiers ; le fake repo (neutre) applique le `ZFilter` de relation et prouve le filtrage ; grep « aucun import backend » dans `z_sub_list_screen.dart`/`z_tabbed_list.dart`.

8. **AC8 — Sémantique de sélection au filtre/relation TRANCHÉE (résout LOW-3).** (a) **Même relation, filtre/recherche utilisateur** : la sélection **persiste par `id`** (une ligne masquée par un filtre puis ré-affichée réapparaît sélectionnée — cohérent E4-4, **documenté**). (b) **Changement de relation parent** (`parentId` change) : `ZSubListScreen.didUpdateWidget` **vide** la sélection (collection différente) ; les onglets à relations différentes ont des sélections **isolées** (jamais partagées). Testable : (a) sélectionner puis filtrer/défiltrer dans la **même** sous-liste → sélection conservée ; (b) reconstruire `ZSubListScreen` avec un **nouveau** `parentId` → sélection **vidée** ; deux onglets → sélections disjointes.

9. **AC9 — a11y onglets (AD-13).** Le `TabBar` expose des `Tab` dont le libellé provient du seam l10n `label(context, tab.labelKey)` (**aucune** chaîne codée en dur) ; l'onglet actif est **annoncé** (`Semantics` sélectionné / rôle onglet) ; cibles tactiles **≥ 48 dp** ; primitives **directionnelles** (`EdgeInsetsDirectional`/`TextAlign.start`, jamais `left/right`). L'a11y des actions/sélection de la sous-liste (boutons `Semantics(button,enabled,label)`, `Semantics(selected:)`) est **préservée** (héritée d'E4-4). Testable : finder `Tab`/`Semantics` (label résolu, onglet actif `selected: true`), contrainte de hauteur ≥ 48, aucun `TextAlign.left`/`EdgeInsets.only(left:)` introduit.

10. **AC10 — SM-5 / neutralité + barrels + vérif verte + non-régression E4.** Les nouveaux fichiers (`z_sub_list_screen.dart`, `z_tabbed_list.dart`, éventuel `z_list_tab.dart`) et la modification de `z_list_controller.dart` importent **uniquement** `package:flutter/{widgets,material,foundation}.dart` + types `zcrud_core` — **zéro** `package:syncfusion`, **zéro** `cloud_firestore`/`firebase`/`hive`, **zéro** gestionnaire d'état, **zéro** dépendance ajoutée (`pubspec.yaml` inchangé, out-degree 0, AD-1). Le chrome des onglets est **pur-Flutter Material** ; les listes passent par le renderer **injecté** (`ZcrudScope.listRenderer`) → testées en layout **`builder`** (SM-5 **exécutable** sans `zcrud_list`). Le **test de graphe SM-5** reste **vert** (`ZSfDataGridRenderer` demeure la seule arête Syncfusion). `baseFilters` est **additif rétro-compatible** (suite E4-3 verte, inchangée). Barrels : `zcrud_core.dart` exporte `ZSubListScreen`/`ZTabbedList`/`ZListTab`. Gardes `presentation_purity`/`no_heavy_file_dep`/`domain_purity`/`style_purity` vertes. `melos run generate` OK → `melos run analyze` RC=0 (14 pkgs) → `flutter test` RC=0 (`zcrud_core` **et** `zcrud_list`) → `melos run verify` RC=0 (CORE OUT=0, SM-5 graphe, secrets, codegen, reflectable, compat).

## Dev Notes

### Décisions tranchées

- **RÉUTILISATION, pas réinvention (cœur de la story).** `ZSubListScreen` et `ZTabbedList` sont des **assembleurs minces** : toute la mécanique de liste (recherche sans accents, filtres/tri, pagination curseur/repli, mapping `T→ZListRow`, 4 vues d'état) vit déjà dans **`ZListController` (E4-3)** + **`DynamicList` (E4-2)** ; les actions/`ZAcl`/sélection/corbeille dans **E4-4** ; le rendu dans le **renderer injecté (E4-1)**. E4-5 **n'ajoute aucune** de ces mécaniques — il les **compose**. Toute tentation de réécrire un moteur de liste, un mapping, une pagination = **STOP**.
- **La relation est un `baseFilters` PERSISTANT, pas un `setFilters`.** `ZListController.setFilters` **remplace** les filtres utilisateur (E4-3). Si la relation parent était poussée via `setFilters`, le premier filtre utilisateur de la sous-liste **effacerait** la relation → fuite de données inter-parents. D'où **`baseFilters`** : socle **immuable** toujours ANDé (`[...baseFilters, ..._filters]`) dans **chaque** requête. Additif, défaut `const []` ⇒ rétro-compatible E4-3. Le **filtre de catégorie** d'un onglet emprunte le **même** mécanisme (`baseFilters` = filtre de catégorie).
- **Préservation d'état des onglets via `AutomaticKeepAliveClientMixin`.** Par défaut, `TabBarView` **détruit** les pages hors écran (perte du `ZListController`/scroll/sélection au switch). Correctif : chaque page enveloppée dans un `_KeepAliveTabPage` (`wantKeepAlive => true`) → le sous-arbre (et son `State` portant les contrôleurs) est **conservé**. C'est l'équivalent « liste » du principe AD-2 (état stable, pas recréé au rebuild). **Ne PAS** stocker l'état d'onglet dans le parent qui se reconstruit ; le laisser vivre dans le `State` keep-alive de chaque page.
- **Sélection : indépendante PAR onglet, keyée par `id` (résout LOW-3).** Chaque onglet a **sa propre** `ZListSelectionController` (sélections disjointes, AC6). Pour la sous-liste seule (AC8) : la sélection **persiste par `id`** au filtre/recherche (cohérent E4-4 — l'état vit hors renderer, keyé par `id` stable), MAIS un **changement de relation parent** (`parentId`) est une **collection différente** ⇒ `didUpdateWidget` **vide** la sélection. Ceci **tranche** le report E4-4 **LOW-3** (« garder = sélection persistante par id ; ou élaguer aux id visibles au refresh ») : **on garde la persistance par id au sein d'une même relation** (pas d'élagage surprenant au simple filtre), **et** on **réinitialise sur changement de relation/onglet** (là où la persistance n'aurait aucun sens). Pas d'élagage automatique aux « lignes visibles » (comportement stable et prévisible ; l'app peut appeler `clearSelection` si elle préfère).
- **Relation neutre `ZFilter(parentField, ZFilterOp.eq, parentId)` (AD-16).** Le contrat `ZDataRequest`/`ZFilter` (E2-2 gelé, `packages/zcrud_core/lib/src/domain/data/z_data_request.dart`) exprime la relation **sans** type backend. L'adaptateur Firestore (E5) traduira `ZFilter(eq)` en `where('parent_id', isEqualTo: …)` (snake_case en persistance) ; ici, un **fake `ZRepository`** in-memory applique les filtres pour prouver le comportement. `parentId` est `Object` (opaque) — le cœur ne présume pas `String` (un id composite reste possible).
- **`ZListTab` porte un `WidgetBuilder`, pas un `ZListController` directement.** Un `ZListController` est un `ChangeNotifier` à cycle de vie (create/dispose) — le passer dans un modèle `const` violerait AD-2. `ZListTab.builder` (`WidgetBuilder`) laisse **chaque page** créer/détenir son contrôleur dans un `State` keep-alive. La catégorisation passe par les `baseFilters` fournis au contrôleur construit dans `builder`.
- **SM-5 : chrome onglets pur-Flutter, listes via renderer injecté.** `TabBar`/`TabBarView` sont Material **pur** (déjà dans `flutter/material.dart`, dépendance légitime du cœur — comme le reste de la couche présentation). Les listes ne rendent Syncfusion **que** si l'app a injecté `ZSfDataGridRenderer` via `ZcrudScope.listRenderer` — sinon layout `builder`/`custom` neutre. Les tests utilisent `builder` → **SM-5 exécutable** : `zcrud_core` reste out-degree 0, `ZSfDataGridRenderer` seule arête Syncfusion.

### Ambiguïtés détectées (à trancher par le dev / signalées)

1. **`baseFilters` : modifier `ZListController` (E4-3) ou composer dans `ZSubListScreen` ?** Retenu : **`baseFilters` sur `ZListController`** (additif, défaut `const []`, rétro-compatible) — c'est le seul point où **toutes** les requêtes (in-memory + backend + `loadMore`) sont émises, donc la seule garantie que la relation **ne fuit jamais**. Alternative rejetée (composer côté `ZSubListScreen` en interceptant `setFilters`) : fragile (le `loadMore`/`refresh` interne rebâtit la requête sans passer par l'intercepteur). **Vérifier** que `_runInMemory` honore aussi la composition.
2. **Résolution LOW-3 (élagage sélection).** Retenu : **persistance par `id` au sein d'une relation** (pas d'élagage au filtre — prévisible, cohérent E4-4) + **réinitialisation sur changement de relation/parent** et **isolation par onglet**. Si l'orchestrateur/PO préfère « élaguer aux id visibles au refresh », le point d'insertion est `ZSubListScreen`/le contrôleur d'onglet (documenté) — mais **non** retenu par défaut (surprise UX du « select-all puis filtre »).
3. **`ZListTab` : `WidgetBuilder` vs modèle typé `{filters, ...}` ?** Retenu : **`WidgetBuilder` + fabrique `ZListTab.category`** — souple (un onglet peut rendre `ZSubListScreen`, une `DynamicList` libre, etc.) tout en offrant le cas courant « catégorie = `baseFilters` ». Un modèle 100 % déclaratif serait moins souple pour les vues custom.
4. **Recherche de la sous-liste : intégrée à `ZSubListScreen` ou laissée à l'app ?** Retenu : **barre de recherche optionnelle** (`showSearch`, défaut à trancher au dev) câblée à `controller.setSearch` (le mini-CRUD est « complet » AC3). L'app peut la désactiver et fournir sa propre UI (le contrôleur reste pilotable).
5. **`didUpdateWidget` recrée-t-il le contrôleur sur changement de `parentId` ?** Retenu : **oui** (nouvelle relation = nouvelles données) + **vide la sélection** (AC8). Le `pageSize`/`schema`/`toRow` sont supposés stables ; si l'app les change dynamiquement, documenter le recyclage.
